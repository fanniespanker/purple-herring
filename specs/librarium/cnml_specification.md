# CNML Specification v0.5.1 (Draft)

This Purple Herring CNML draft incorporates the current CNML structural-frame, addressing/milestone, discourse, linguistic-system, prosody, affect, C4 integration, and information-integrity revisions.

## Foundations

CNML (Contextual Narrative Markup Language) is a constrained XML vocabulary with semantic rules.

All CNML documents MUST be well-formed XML.

CNML is primarily an experiential and discourse-oriented representation system.

Experiential content refers to represented subjective or interpretive states, not objective truth claims.

CNML encodes semantic, experiential, structural, temporal, prosodic, and spatial relationships, not canonical presentation.

Evaluation, rendering, playback, and projection are interpreter-dependent realizations of CNML source structure.

Operational interpretation MAY derive additional structure for rendering or execution purposes, but such derived structure MUST NOT be treated as intrinsic source semantics.

---

## Core Evaluation Model

CNML evaluation operates across four primary expressive planes:

```text
Narrative × Temporal × Prosodic × Spatial
```

These planes are independently representable and compositional.

## Planes

| Plane     | Meaning                          |
|-----------|----------------------------------|
| Narrative | discourse, cognition, experience |
| Temporal  | musical/performance progression  |
| Prosodic  | affective modulation             |
| Spatial   | typographic/spatial arrangement  |

---

## Discourse Epistemics

CNML discourse does not represent objective reality.

Discourse content is interpreted as a projection generated by a narrator, speaker, experiencer, evaluator, or other discourse context.

Discourse contexts MAY be:

- unreliable
- dishonest
- misleading
- hallucinating
- delusional
- temporally inconsistent
- partially reconstructive

CNML discourse is therefore epistemically unstable by default.

## Structural Frames

Structural frames partition CNML source content into document, publication, matter, and evaluation scopes.

A structural frame identifies a source region and establishes containment.

A structural frame MUST NOT be interpreted, by itself, as asserting:

- in-world truth
- chronological order
- narrative reliability
- experiential modality
- semantic equivalence
- publication-resource identity
- rendering style

Such meanings belong to explicit discourse, experiential, semantic, publication, rendering, or external graph constructs.

### Core Structural Frames

CNML defines the following structural frame elements:

```text
cnml
series
story
book
frontmatter
mainmatter
backmatter
part
act
chapter
scene
section
```

### Structural Frame Roles

`<cnml>` is the root CNML container and establishes the document-level CNML processing context.

`<series>` is a publication-level container grouping one or more work-level containers.

`<story>` is a work-level publication frame for short-form or story-structured works.

`<book>` is a primary long-form work-level container.

`<frontmatter>` contains material preceding the main matter.

`<mainmatter>` contains the main work-content matter frame and the primary progression sequence of the work.

`<backmatter>` contains material following the main matter.

`<part>` is an optional high-level subdivision of `<mainmatter>`. It is above ordinary progression units such as acts, chapters, and scenes. It is not a generic segmentation element. Use `<part>` when the main matter has major authorial divisions above acts or chapters, such as named parts, books-within-books, movements, large thematic divisions, or macrosections.

`<act>` is a dramatic or authorial macro-division within `<mainmatter>` or `<part>`.

`<chapter>` is a major work subdivision within `<mainmatter>`, `<part>`, or `<act>`.

`<scene>` is a local source subdivision within `<mainmatter>`, `<part>`, `<act>`, or `<chapter>`.

`<section>` is a paratextual structural subdivision for `<frontmatter>` and `<backmatter>`.

### Publication Metadata Elements

Publication frames MAY contain publication metadata elements.

Publication frames SHOULD contain at least one of each of the following when applicable:

- `<title>`
- `<author>`
- `<date of="publication">`
- `<identifier scheme="identifier_scheme">`
- `<synopsis>`
- `<license>` — license information, which MUST contain one `<name>` element, and MAY contain any number of each of the following:
  - `<identifier scheme="identifier_scheme">` — an identifier, such as a URI/IRI, for the license;
  - `<code>` — the legal code of the license;
  - structured data that represents the terms of the license.

Publication frames MAY also contain any number of genre and audience elements:

- `<genre>`
- `<audience>`

Publication frames MAY also contain any number of subject elements which indicate the depth of those subjects' presence and/or the author's position, whether moral, academic, or otherwise, regarding those subjects:

- `<references>`
- `<depicts>`
- `<explores>`
- `<examines>`
- `<challenges>`
- `<opposes>`
- `<parodies>`

Publication frames MAY also embed or reference bibliographic data from other works through the `<external>` element, which contains `<work>` elements.

Example:

```xml
<book vocabulary="https://github.com/fanniespanker/Media-Description-Ontology/0.1/ns">
  <title>Hardboiled Whore</title>
  <author>Fannie Spanker</author>
  <date of="publication">2025</date>
  <identifier scheme="IRI">https://fanniespanker.com/armedescorts/hardboiledwhore/</identifier>
  <license>
    <name>Creative Commons Attribution Noncommercial ShareAlike 4.0 International</name>
    <iri>https://creativecommons.org/licenses/by-nc-sa/4.0/legalcode</iri>
  </license>

  <synopsis style="diegetic">
    <heading>Some cops love themselves. Some cops know they’re human garbage.</heading>
  </synopsis>

  <genre resource="Fiction.humor/sex_comedy">sex comedy</genre>
  <parodies resource="#Columbo">Columbo</parodies>
  <supports resource="Politics.movements/labor/unions/activism">labor unions in bodily autonomy-oriented feminist activism</supports>
  <audience resource="People.marginalized_groups/GSM">gender and sexual minorities</audience>

  <external>
    <work
        about="#Columbo"
        medium="Media.medium/television"
        format="Media.format/series?of=Media.format/movie"
        resource="https://www.imdb.com/title/tt0061496"
    >
      <title>Columbo</title>
      <creator>Richard Levinson</creator>
      <creator>William Link</creator>
      <genre resource="Fiction.crime/mystery/procedural">police procedural mystery</genre>

      <character resource="https://id.loc.gov/authorities/sh93004873">
        <name>Columbo</name>
      </character>
    </work>
  </external>

  <mainmatter>
    <chapter n="Chapter 1">
      <scene n="Cheap Room">
        <p>...</p>
      </scene>
    </chapter>
  </mainmatter>
</book>
```

### Structural Container Rules

Each structural frame has a defined set of legal immediate containers.

A structural frame is conforming when its immediate parent is one of its legal containers and the containing context satisfies Same-Context Rank Discipline.

The root `<cnml>` MAY contain any structural frame as an immediate child. This permits CNML documents to represent structural fragments such as a standalone chapter, scene, or section.

A structural frame appearing directly under `<cnml>` is interpreted as the root frame of the current CNML fragment. Missing ancestors are not implied unless supplied by metadata, evaluator context, or an external referencing layer.

| Structural frame | Legal immediate containers |
|---|---|
| `<series>` | `<cnml>` |
| `<story>` | `<cnml>`, `<series>` |
| `<book>` | `<cnml>`, `<series>` |
| `<frontmatter>` | `<cnml>`, `<story>`, `<book>` |
| `<mainmatter>` | `<cnml>`, `<story>`, `<book>` |
| `<backmatter>` | `<cnml>`, `<story>`, `<book>` |
| `<part>` | `<cnml>`, `<mainmatter>` |
| `<act>` | `<cnml>`, `<mainmatter>`, `<part>` |
| `<chapter>` | `<cnml>`, `<mainmatter>`, `<part>`, `<act>` |
| `<scene>` | `<cnml>`, `<mainmatter>`, `<part>`, `<act>`, `<chapter>` |
| `<section>` | `<cnml>`, `<frontmatter>`, `<backmatter>`, `<section>` |

### Same-Context Rank Discipline

A context MUST NOT mix structural frames of different hierarchy ranks.

A structural frame may appear directly in a context only when no higher-rank structural frame appears in that same context.

If a context contains a higher-rank frame, lower-rank frames and block content MUST appear inside the appropriate descendant frame rather than as siblings of the higher-rank frame.

This permits structural fragments while preserving the CNML frame hierarchy.

Valid fragment:

```xml
<cnml>
  <chapter n="Chapter 4">
    ...
  </chapter>
</cnml>
```

Invalid mixed context:

```xml
<cnml>
  <book n="Hardboiled Whore">
    ...
  </book>
  <chapter n="Chapter 4">
    ...
  </chapter>
</cnml>
```

Valid hierarchical context:

```xml
<cnml>
  <book n="Hardboiled Whore">
    <mainmatter>
      <chapter n="Chapter 4">
        ...
      </chapter>
    </mainmatter>
  </book>
</cnml>
```

### Mainmatter Structural Sequence

The elements `<part>`, `<act>`, `<chapter>`, and `<scene>` form the mainmatter structural sequence.

They represent authored/work-level progression inside the main content of a `<book>` or `<story>`.

The rank order is:

```text
part → act → chapter → scene
```

### Sections

Sections are exclusively for structuring `<frontmatter>` and `<backmatter>`.

Sections MAY nest within `<frontmatter>`, `<backmatter>`, or another `<section>` that is itself contained within frontmatter or backmatter.

```xml
<backmatter>
  <section n="notes">
    <section n="note-1">
      ...
    </section>
  </section>
</backmatter>
```

The `role` attribute is optional and evaluator-defined.

Examples:

- dedication
- acknowledgments
- epigraph
- commentary
- notes
- appendix
- index

### Frame Headers

The following structural frames MAY contain zero or one of each frame-header element permitted by the active CNML profile:

```text
series
story
book
section
part
act
chapter
scene
```

Common frame-header elements include:

```text
title
synopsis
subheading
```

Example:

```xml
<part n="Part 1">
  <title>The City Eats Its Young</title>
  <chapter n="Chapter 1">
    ...
  </chapter>
</part>
```

The `n` attribute and `<title>` serve different roles. `n` supplies an address segment; `<title>` supplies title/display metadata.

### Content Containers

Structural-frame parentage defines where structural frames may appear.

Text-bearing source content is represented by block content and inline content, not by structural frames.

Block content includes paragraphs, expression elements, experiential modality elements, poem/music blocks, semantic extension points, and other source-content elements permitted by the active CNML profile.

Block content MAY appear inside:

```text
cnml
frontmatter
mainmatter
backmatter
part
act
chapter
scene
section
```

A block-content element appearing directly under `<cnml>` is interpreted as a content fragment.

Block content is lower-rank than structural frames for Same-Context Rank Discipline.

If a context contains a structural frame, block content MUST appear inside the appropriate descendant frame rather than as a sibling of that structural frame.

Raw character data belongs inside text-bearing block or inline elements.

Inline content includes text nodes, milestones, information-integrity elements, prosody elements, and other inline elements permitted by the active CNML profile.

Structural frames SHOULD NOT contain raw character data directly except for insignificant whitespace.

Text-bearing block and inline elements define their own mixed-content behavior.

### Narrative Voice Context

CNML defines `narrator` and `voice` as optional inheritable discourse-context attributes.

These attributes MAY appear on structural frames and discourse containers.

`narrator` identifies the discourse source, narrator identity, persona, or narrative agent associated with the content.

`voice` identifies narrative voice, register, style, tone, rhetorical mode, or discourse channel.

`narrator` and `voice` are inherited by descendant content unless overridden by a nearer descendant value.

Inheritance is nearest-ancestor inheritance:

```text
effective(narrator, node) = nearest ancestor-or-self value of @narrator
effective(voice, node)    = nearest ancestor-or-self value of @voice
```

`narrator` and `voice` SHOULD NOT be conflated. `narrator` identifies who or what narrates; `voice` identifies how the discourse is voiced.

Expression-specific attributes such as `who` identify represented speakers, thinkers, feelers, or experiencers. They do not replace inherited narration context unless the expression element also explicitly sets `narrator`.

Example:

```xml
<book n="Hardboiled Whore" narrator="Andrea" voice="retrospective">
  <mainmatter>
    <chapter n="Chapter 1">
      <scene n="Cheap Room" voice="hardboiled">
        <p>This inherits narrator Andrea and voice hardboiled.</p>
        <say who="Modo">This is spoken by Modo.</say>
      </scene>
    </chapter>
  </mainmatter>
</book>
```

### Discourse Content

Primary experiential discourse MAY appear directly inside mainmatter structural frames.

Discourse context is represented through structural containment, modality elements, expression elements, and inheritable discourse-context attributes such as `narrator` and `voice`.

Example:

```xml
<scene n="Cheap Room" narrator="Andrea" voice="hardboiled">
  <p>She woke up with the city chewing on her name.</p>
  <say who="Modo">Don’t be alarmed.</say>
  <remember>
    <p>The knife had been clean yesterday.</p>
  </remember>
</scene>
```

## Addressing, Milestones, and Derived Names

This section defines CNML-local addressing over authored `n` values, milestones, span expressions, grouped spans, derived address segments, and resource-internal subresource path behavior shared with Purple Herring where applicable.

### Local CNML Names

CNML elements MAY carry an `n` attribute.

The `n` attribute defines an authored local CNML name usable as a segment in CNML address expressions.

```xml
<book n="Hardboiled Whore">
  <act n="Act 1">
    <chapter n="Chapter 4">
      <p>
        <ms n="ms5"/>She remembered too late.<ms n="ms9"/>
      </p>
    </chapter>
  </act>
</book>
```

The milestone may be addressed as:

```text
Hardboiled Whore.Act 1.Chapter 4.ms5
```

`n` is not an XML identifier. It is not required to be document-global.

`xml:id` MAY additionally be used when XML-native document-global identity is required.

```xml
<ms n="ms5" xml:id="hw-a1-c4-ms5"/>
```

`resource` MAY be used separately for persistent semantic or publication identity.

```xml
<chapter n="Chapter 4" resource="https://example.org/work/hw/chapter/4">
```

Thus:

| Attribute | Meaning |
|---|---|
| `n` | CNML local address segment |
| `xml:id` | XML document-global node identity |
| `resource` | persistent semantic/publication resource identity |

### CNML Address Expressions

A CNML address expression is a path over authored `n` values and/or processor-derived address segments.

The path separator is `.`.

```text
Hardboiled Whore.Act 1.Chapter 4.ms5
```

Each unescaped `.` separates one path segment from the next.

A path segment resolves against an addressable CNML unit in the current address scope.

```ebnf
address = segment, { ".", segment } ;
```

CNML address expressions are commonly written inside XML attribute values, which already provide the outer quotation layer.

```xml
<annotation target="Hardboiled Whore.Act 1.Chapter 4.ms5"/>
```

Whitespace MAY appear literally in `n` values and in CNML address expressions.

### Escaping in Address Expressions

Characters or sequences with syntactic meaning in CNML address expressions MUST be escaped when they occur literally inside an `n` value.

CNML address expressions use percent escapes.

Reserved syntax:

```text
.      path separator
--     span separator
%      escape introducer
( )    grouping delimiters
```

Required escapes include:

```text
.   => %2E
%   => %25
(   => %28
)   => %29
-   => %2D when needed to prevent `--`
```

Percent escapes are interpreted during address-expression parsing, after XML entity processing and before segment comparison.

Example:

```xml
<ms n="ms 5.5"/>
```

Address:

```text
ms 5%2E5
```

Example with a full path:

```xml
<book n="Hardboiled Whore">
  <act n="Act 1">
    <chapter n="Chapter 4">
      <ms n="ms 5.5"/>
    </chapter>
  </act>
</book>
```

Address:

```text
Hardboiled Whore.Act 1.Chapter 4.ms 5%2E5
```

Authors SHOULD prefer stable, readable `n` values. Human-readable titles MAY be encoded as `n` values, but titles and display labels MAY also be represented separately.

```xml
<chapter n="chapter-4">
  <title>Chapter 4</title>
</chapter>
```

### Milestones

CNML provides the empty `<ms/>` element as a source-position milestone.

A milestone marks an addressable position in an ordered CNML source stream.

```xml
<ms n="ms5"/>
```

A milestone:

1. MUST be empty.
2. MAY carry an authored `n`.
3. MAY carry an `xml:id` for XML-native identity.
4. Marks a position, not a content container.
5. Has no default rendering.
6. Does not itself define a span.
7. Is semantically neutral unless referenced by another construct.

Example:

```xml
<p>
  <ms n="start"/>The old house <emph>remembered</emph> her.<ms n="end"/>
</p>
```

The milestones `start` and `end` mark source positions surrounding the sentence.

Milestones MUST NOT produce visible output by default.

Renderers MAY expose milestones in debugging, editorial, or annotation views.

### Span Expressions

A CNML span expression identifies an ordered source interval between two addresses.

The span separator is `--`.

```text
start-address--end-address
```

Example:

```text
Hardboiled Whore.Act 1.Chapter 4.ms5--Hardboiled Whore.Act 1.Chapter 4.ms9
```

A milestone-delimited span contains all source content after the start milestone and before the end milestone.

The milestone elements themselves are boundary markers and are not part of the span payload.

Example:

```xml
<p>
  <ms n="a"/>hello<ms n="b"/>
</p>
```

Span:

```text
a--b
```

Payload:

```text
hello
```

Within a single source stream, the start address of a span MUST precede or equal the end address.

If both addresses resolve to the same source position, the resulting span is empty.

Reverse spans are not defined by CNML alpha.

### Grouped Shared-Prefix Spans

A span whose endpoints share a prefix MAY be written using grouped shared-prefix syntax.

```text
prefix.(start--end)
```

Example:

```text
Hardboiled Whore.Act 1.Chapter 4.(ms5--ms9)
```

This is equivalent to:

```text
Hardboiled Whore.Act 1.Chapter 4.ms5--Hardboiled Whore.Act 1.Chapter 4.ms9
```

The expression inside parentheses is evaluated relative to the shared prefix.

Grouped spans are authoring sugar. Processors MAY normalize them to expanded span expressions.

For alpha, grouped spans MUST NOT contain nested grouped expressions.

Valid:

```text
Book.Act 1.Chapter 4.(ms5--ms9)
```

Invalid in alpha:

```text
Book.(Act 1.(Chapter 4.(ms5--ms9)))
```

### Overlapping XML Structure

Milestone-delimited spans MAY overlap XML element boundaries.

This permits CNML to address source intervals that cannot be represented by ordinary XML containment without altering the document tree.

Example:

```xml
<p>
  She <ms n="a"/>almost <emph>remembered<ms n="b"/> everything</emph>.
</p>
```

The span `a--b` begins in plain text and ends inside `<emph>`.

### Structural Boundary Constraint

A locally conforming milestone-delimited span SHOULD NOT cross structural frame boundaries unless a referencing layer explicitly defines cross-frame span semantics.

A milestone pair is locally conforming when both milestones occur within the same nearest structural frame.

Structural frames include:

```text
series
story
book
frontmatter
mainmatter
backmatter
section
part
act
chapter
scene
```

Example, locally conforming:

```xml
<chapter n="Chapter 4">
  <p><ms n="a"/>Beginning...</p>
  <p>...ending<ms n="b"/></p>
</chapter>
```

Example, not locally conforming in alpha:

```xml
<chapter n="Chapter 4">
  <p><ms n="a"/>Beginning...</p>
</chapter>
<chapter n="Chapter 5">
  <p>...ending<ms n="b"/></p>
</chapter>
```

Referencing layers MAY define cross-frame behavior, but CNML alpha processors are not required to support it.

### Address Scopes

CNML addresses are evaluated relative to an address scope.

Structural elements with `n` values establish addressable path segments.

The following elements SHOULD be treated as addressable units when present:

```text
series
story
book
frontmatter
mainmatter
backmatter
section
part
act
chapter
scene
poem
music
stanza
line
p
s
w
ms
```

A complete path descends through addressable units.

Example:

```xml
<book n="Hardboiled Whore">
  <act n="Act 1">
    <chapter n="Chapter 4">
      <p n="opening">
        <ms n="ms5"/>Text.<ms n="ms9"/>
      </p>
    </chapter>
  </act>
</book>
```

Addresses:

```text
Hardboiled Whore.Act 1.Chapter 4
Hardboiled Whore.Act 1.Chapter 4.opening
Hardboiled Whore.Act 1.Chapter 4.opening.ms5
```

Relative addressing MAY be supported by referencing layers, but CNML alpha only requires absolute or context-provided path resolution.

### Derived Address Segments

Any addressable CNML unit without an authored `n` MAY be assigned a processor-derived address segment.

Derived address segments MUST begin with `_`.

Derived segments are non-authorial. They are not intrinsic source data.

Example:

```xml
<book n="Hardboiled Whore">
  <act>
    <chapter>
      <p>First sentence. Second sentence.</p>
    </chapter>
    <chapter n="The Cheap Room">
      <p>Another paragraph.</p>
    </chapter>
  </act>
</book>
```

Possible derived and explicit addresses:

```text
Hardboiled Whore._act1
Hardboiled Whore._act1._ch1
Hardboiled Whore._act1._ch1._p1
Hardboiled Whore._act1._ch1._p1._s1
Hardboiled Whore._act1._ch1._p1._s2

Hardboiled Whore._act1.The Cheap Room
Hardboiled Whore._act1.The Cheap Room._p1
```

Authored `n` values beginning with `_` are reserved and SHOULD NOT be used in conforming CNML documents.

Explicit `n` values define authored address segments and take precedence over derived address segments.

### Derived Segment Vocabulary

Derived structural segments SHOULD use stable element-kind abbreviations followed by a one-based ordinal within the nearest relevant address scope.

Recommended derived segment forms:

```text
_series1       first series
_story1        first story
_book1         first book
_frontmatter1  first frontmatter
_mainmatter1   first mainmatter
_backmatter1   first backmatter
_sec1          first section
_part1         first part
_act1          first act
_ch1           first chapter
_sc1           first scene
_p1            first paragraph
_l1            first line
_st1           first stanza
_s1            first sentence
_w1            first word
_t1            first token
_ms1           first anonymous or generated milestone
```

Element-kind ordinals are counted by element kind within the relevant containing scope.

Example:

```xml
<act>
  <chapter/>
  <interlude/>
  <chapter/>
</act>
```

Derived addresses:

```text
_act1._ch1
_act1._interlude1
_act1._ch2
```

not:

```text
_act1._ch1
_act1._interlude2
_act1._ch3
```

The chapter count counts chapters, not all child elements.

### Derived Address Stability

Derived address segments are positional and may change when the source changes.

Structural derived addresses are moderately stable:

```text
Hardboiled Whore._act1._ch4
```

but may change if a new act or chapter is inserted before them.

Lexical derived addresses are volatile:

```text
Hardboiled Whore._act1._ch4._p3._s2._w5
```

Sentence, word, and token segmentation are profile-dependent unless explicitly marked in source.

Processors MAY expose sentence, word, or token addresses only under a declared segmentation profile.

Milestones SHOULD be used for durable fine-grained addressing.

Example:

```xml
<p>
  She <ms n="knife-start"/>remembered<ms n="knife-end"/> too late.
</p>
```

Durable span:

```text
knife-start--knife-end
```

or with shared prefix:

```text
Chapter 4.(knife-start--knife-end)
```

### Resource and Subresource Paths

CNML address expressions may be used as subresource paths inside larger resource references.

A resource identity and a path inside that resource SHOULD remain separable.

Recommended conceptual form:

```text
resource#subresource-path
```

Example:

```text
Hardboiled Whore#Act 1.Chapter 4.ms5
Hardboiled Whore#Act 1.Chapter 4.(ms5--ms9)
```

Within Purple Herring or another semantic layer, a CNML subresource path SHOULD be embedded or quoted as a resource-internal path rather than reinterpreted as an ontology path.

Example:

```text
cnml:"Hardboiled Whore#Act 1.Chapter 4.(ms5--ms9)"
```

CNML and Purple Herring SHOULD share the subresource path grammar where both are addressing positions inside concrete resources.

This does not require CNML source paths and PH semantic/concept traversal paths to have identical semantics.

### Summary of Address Syntax

```text
.      path/subresource separator
--     span separator
(...)  grouped shared-prefix expression
%XX    percent escape
_      derived segment prefix
#      resource/subresource boundary, where used by a referencing layer
```

Examples:

```text
Hardboiled Whore.Act 1.Chapter 4.ms5

Hardboiled Whore.Act 1.Chapter 4.ms5--Hardboiled Whore.Act 1.Chapter 4.ms9

Hardboiled Whore.Act 1.Chapter 4.(ms5--ms9)

Hardboiled Whore.Act 1.Chapter 4.ms 5%2E5

Hardboiled Whore._act1._ch4._p2._s1

Hardboiled Whore#Act 1.Chapter 4.(ms5--ms9)
```

### Addressing Alpha Constraints and Open Questions

For alpha:

1. `n` is the authored CNML address-segment attribute.
2. `<ms/>` is the canonical zero-width source-position milestone.
3. `.` separates address segments.
4. `--` separates span endpoints.
5. `(...)` compresses shared-prefix spans.
6. `%XX` escapes reserved characters inside address expressions.
7. `_` prefixes derived, non-authorial address segments.
8. `xml:id` remains optional and XML-native; it does not replace CNML path addressing.
9. Derived sentence, word, and token addresses require declared segmentation behavior before they can be considered reproducible.
10. Cross-frame spans are outside required alpha processor behavior unless a referencing layer defines them.

Open questions for later versions:

1. Whether relative path forms should exist.
2. Whether grouped shared-prefix syntax should allow nesting after alpha.
3. Whether address expressions should support selection/filter predicates.
4. Whether derived-address profiles should be declared globally, per book, per chapter, or per processing trace.
5. Whether anonymous milestones should receive `_msN` derived names by default or only under debugging/annotation profiles.

## Experiential Modalities

### Modality Elements

```xml
<dream>
<fantasy>
<flashback>
<remember>
```

These elements form a 2×2 cognitive operator grid.

### Cognitive Axes

#### Epistemic Source

| Value | Meaning |
|---|---|
| irrealis | generated experiential content |
| memory | retrieved experiential content |

#### Volition

| Value | Meaning |
|---|---|
| involuntary | non-deliberate emergence |
| voluntary | deliberate invocation |

#### Mapping

| Element | Source | Volition |
|---|---|---|
| dream | irrealis | involuntary |
| fantasy | irrealis | voluntary |
| flashback | memory | involuntary |
| remember | memory | voluntary |

### Formal Model

```text
NarrativeMode = EpistemicSource × Volition
```

## CNML-Native Semantic Relations and Embedded Data

CNML distinguishes CNML-native C4 relation blocks from generic embedded data blocks.

`<c4>` is tightly coupled to CNML and receives CNML-aware resource access semantics.

`<data lang="...">` embeds generic non-CNML-native data or relation languages. Such blocks are parsed according to their declared language and do not receive C4/CNML resource access semantics by default unless an evaluator or adapter explicitly defines such mapping.

### C4 Blocks

```xml
<c4>
```

`<c4>` embeds C4 / Purple Herring relation statements whose resource-resolution context is the surrounding CNML document.

A `<c4>` block MAY contain one or more C4 fish statements. The block itself is not necessarily a single fish or a whole school. Evaluators MAY parse a `<c4>` block into a C4 school fragment.

A `<c4>` block MAY address CNML-native resources such as:

- the current structural frame resource;
- publication frames such as `<series>`, `<story>`, and `<book>`;
- matter and mainmatter frames such as `<frontmatter>`, `<mainmatter>`, `<backmatter>`, `<part>`, `<act>`, `<chapter>`, and `<scene>`;
- explicitly selected ancestor elements such as expression modalities, prosody elements, integrity elements, and other addressable CNML elements;
- declared anchors or milestone spans;
- `xml:id`-addressable elements;
- evaluator-defined CNML projections.

#### Current Structural Frame Resource

Inside `<c4>`, `#` denotes the current structural frame resource.

The current structural frame resource is the nearest enclosing ancestor CNML element belonging to the structural-frame set.

The structural-frame set is:

```text
series
story
book
frontmatter
mainmatter
backmatter
section
part
act
chapter
scene
```

By default, `#` MUST NOT resolve to discourse, expression, prosody, integrity, data, or other non-structural elements merely because the `<c4>` block is nested inside them.

Example:

```xml
<book>
  <mainmatter>
    <chapter>
      <scene>
        <say affect="anger">
          <quiet>
            <c4>
              #&has_mood@Affect/dread;
            </c4>
          </quiet>
        </say>
      </scene>
    </chapter>
  </mainmatter>
</book>
```

In this example, `#` resolves to the nearest structural frame, `<scene>`, not to `<quiet>` or `<say>`.

#### Explicit Ancestor Selector

The explicit ancestor selector has the form:

```text
+{ancestor}#
```

`+{ancestor}#` overrides default `#` resolution by walking upward from the current `<c4>` block to the nearest enclosing ancestor whose element name or evaluator-recognized ancestor type is `{ancestor}`.

The `{ancestor}` term is an ancestor type, not necessarily a structural frame. It MAY name structural frames, experiential modalities, expression modalities, prosody elements, integrity elements, data elements, or other addressable CNML element types when evaluator support exists.

If no matching ancestor exists, resolution fails unless an evaluator-defined fallback policy is declared.

Examples:

```xml
<c4>
  #&has_mood@Affect/dread;
  +book#&supports@Politics/movements/feminism;
  +scene#&has_mood@Affect/dread;
  +say#&has_affect@Affect/anger;
  +quiet#&has_quality@Delivery/controlled;
</c4>
```

In this model:

```text
#        = nearest enclosing structural frame
+book#   = nearest enclosing <book>
+scene#  = nearest enclosing <scene>
+say#    = nearest enclosing <say>
+quiet#  = nearest enclosing <quiet>
```

Authors SHOULD use bare `#` when referring to the current structural frame, and `+{ancestor}#` when referring to a non-default ancestor or to a more distant structural ancestor.

C4 resource access semantics are CNML-integrated. They are not equivalent to RDF URI expansion, JSON-LD context processing, Turtle prefix handling, or other external relation-format semantics.

### Generic Data Blocks

```xml
<data lang="...">
```

`<data>` embeds external data, metadata, relation languages, or serialization formats. The `lang` attribute is REQUIRED and identifies the embedded language or format.

Examples:

```xml
<data lang="turtle">
  @prefix ex: <https://example.org/> .
  ex:scene ex:affect ex:dread .
</data>

<data lang="json-ld">
{
  "@context": "https://schema.org",
  "@type": "Book",
  "name": "Hardboiled Whore"
}
</data>

<data lang="rdfxml">
  <!-- RDF/XML content -->
</data>

<data lang="ntriples">
  <https://example.org/scene> <https://example.org/affect> <https://example.org/dread> .
</data>
```

Recommended `lang` values include, but are not limited to:

```text
c4
turtle
json-ld
rdfxml
ntriples
nquads
trig
owl-functional
xml
json
yaml
```

`<data lang="c4">` MAY be accepted as a generic embedded-data form, but it is not equivalent to `<c4>` unless an evaluator explicitly grants it the same CNML-native resource access semantics. Authors SHOULD use `<c4>` when C4 is intended to operate as the CNML-native semantic relation layer.

## Expression Modalities

Expression modalities mark the kind of discourse-content being represented. They mark what communicative or experiential channel carries the content, not necessarily the literal surface action that caused it.

```xml
<say lang="..." system="...">
<think>
<feel>
<express with="...">
```

### Semantics

| Element | Meaning |
|---------|---------|
| `<say>` | externalized linguistic discourse |
| `<think>` | propositional/internal language |
| `<feel>` | autonomic, affective, limbic, or embodied cognition |
| `<express with="...">` | semantic content conveyed through nonlinguistic expressive channels such as body, hands, face, gaze, posture, stillness, or silence |

`<say>` SHOULD be used for externalized linguistic discourse, including spoken language, written language, signed language, and other language-bearing systems. The `lang` and `system` attributes identify the represented language and the surface notation, script, transcription system, or encoding used in the element content.

Examples:

```xml
<say lang="en" system="Latn">I need help.</say>

<say lang="el" system="Grek">Χρειάζομαι βοήθεια.</say>

<say lang="ru" system="Cyrl">Мне нужна помощь.</say>

<say lang="ja" system="Jpan">助けてください。</say>

<say lang="zh-Hant" system="Hant">我需要幫助。</say>

<say lang="chr" system="Cher">...</say>

<say lang="ase" system="gloss">I NEED HELP</say>

<say lang="ase" system="Sgnw">...</say>

<say lang="ase" system="hamnosys">...</say>

<say lang="bfi" system="stokoe">...</say>

<say lang="ase-x-family-home-sign" system="prose">come here</say>
```

`lang` SHOULD use BCP 47-style language tags where possible. Dialects, regions, scripts, and variants SHOULD be encoded using standard subtags where available, and MAY use private-use subtags when no standard tag exists.

Examples:

```xml
<say lang="en-US" system="Latn">...</say>
<say lang="en-GB" system="Latn">...</say>
<say lang="yue-Hant-HK" system="Hant">...</say>
<say lang="nan-Hant-TW" system="Hant">...</say>
<say lang="ca-valencia" system="Latn">...</say>
<say lang="en-x-appalachian" system="Latn">...</say>
```

`system` SHOULD use ISO 15924 script identifiers where applicable. When the content is encoded using a notation, transcription system, glossing convention, sign-language writing system, or CNML-specific DSL not covered by ISO 15924, `system` MAY use an evaluator-defined token.

Recommended ISO 15924-style `system` values include, but are not limited to:

```text
Latn
Grek
Cyrl
Arab
Hebr
Deva
Beng
Guru
Gujr
Taml
Telu
Knda
Mlym
Sinh
Thai
Laoo
Tibt
Mymr
Geor
Armn
Ethi
Cher
Hani
Hans
Hant
Hira
Kana
Jpan
Kore
Hang
Sgnw
```

Recommended non-ISO / notation-oriented `system` values include, but are not limited to:

```text
gloss
hamnosys
stokoe
signscript
prose
cnml-sign
```

For signed languages, facial expression, gaze, posture, and body movement that are grammatical features of the signed language are interpreted according to the declared `system`. They SHOULD NOT be separately represented with `<express>` unless the author intends an additional nonlinguistic expression outside the signed utterance.

`<express>` SHOULD be used for semantic content communicated through nonlinguistic expressive channels.

Examples:

```xml
She curled her finger, <express with="finger">come hither</express>.

She stared at me, <express with="eyes">Are you kidding me?</express>.

He shrugged, <express with="shoulders">I don’t know.</express>.
```

Physical description MAY remain ordinary prose around the expression modality. Evaluators MUST NOT assume that the prose description and tagged discourse-content are identical semantic structures.

### `<express>` Channel Vocabulary

The `with` attribute identifies one or more qualitative expression channels.

`with` is a whitespace-separated token list. Multiple tokens indicate that the content is conveyed through a composite expression involving all listed channels. Token order SHOULD NOT be semantically significant unless an evaluator explicitly defines ordered interpretation.

Example:

```xml
<express with="arms hands">I surrender!</express>
```

Recommended core `with` tokens:

```text
face
eyes
gaze
mouth
brows
head
hand
hands
finger
arms
shoulders
posture
body
movement
stillness
silence
```

These tokens are author-facing channel descriptors, not a closed anatomical registry. Evaluators MAY accept additional qualitative tokens.

`body` is the general fallback channel. Authors SHOULD prefer a more specific token when the expressive channel is clear.

Separate simultaneous meanings SHOULD be represented with separate `<express>` elements rather than overloading a single composite-channel expression.

Example:

```xml
<express with="hands">I surrender.</express>
<express with="eyes">But I hate you.</express>
```

### Affect Attribute

Expression modality elements MAY carry an `affect` attribute.

```xml
<say affect="...">
<think affect="...">
<feel affect="...">
<express with="..." affect="...">
```

`affect` marks the affective components coloring or charging the represented expression, thought, feeling, or nonlinguistic expression. It is not identical to prosody: affect describes emotional composition; prosody describes performed quality, delivery, articulation, or timing.

`affect` is a whitespace-separated token list of noun-form affect components. Multiple tokens indicate co-present affective components rather than ordered sequence, ranking, or intensity. Token order SHOULD NOT be semantically significant unless an evaluator explicitly defines ordered interpretation.

Affect tokens SHOULD be nouns or noun-like nominal forms, not adjectives. Authors SHOULD prefer:

```text
anger
joy
grief
resentment
tenderness
nostalgia
homesickness
protectiveness
love
dread
shame
desire
relief
```

over adjective forms such as:

```text
angry
joyful
grief-stricken
resentful
tender
nostalgic
homesick
protective
loving
dreadful
ashamed
desiring
relieved
```

Examples:

```xml
<say affect="anger grief">Don’t touch me.</say>

<think affect="nostalgia shame">I used to love this place.</think>

<express with="eyes" affect="disbelief irritation">Are you kidding me?</express>

<say affect="homesickness protectiveness love">Stay where I can see you.</say>

<feel affect="dread desire">a pull in the stomach</feel>
```

Multiword affect labels SHOULD be represented with a single token only when the author intends a lexicalized affect category, for example:

```xml
<feel affect="separation_anxiety">a cold pull under the ribs</feel>
```

Otherwise, authors SHOULD compose affect from separate noun tokens.

## Prosody

Prosody is a compositional modulation system. Prosodic elements modify expression quality, articulation, phonation, or performed timing. Prosody does not alter discourse structure.

Prosody:

- modifies expression quality;
- does not alter discourse structure;
- composes recursively;
- is evaluator-interpreted.

Prosodic elements MAY also carry `affect` when the affective coloring applies specifically to the prosodic modifier rather than the whole expression.

Example:

```xml
<say affect="anger"><quiet affect="control">Get out.</quiet></say>
```

### Dimensional Prosody

Dimensional prosody consists of opposed, composable expressive-quality dimensions.

```xml
<quiet>     <loud>
<calm>      <excited>
<gentle>    <forceful>
<smooth>    <harsh>
<engaged>   <detached>
<emph>      <deemph>
```

| Pair | Axis |
|------|------|
| `<quiet>` / `<loud>` | volume, projection, acoustic intensity |
| `<calm>` / `<excited>` | arousal |
| `<gentle>` / `<forceful>` | affective pressure, interpersonal force |
| `<smooth>` / `<harsh>` | delivery texture |
| `<engaged>` / `<detached>` | affective or interpersonal involvement |
| `<emph>` / `<deemph>` | stress or de-stress |

`<emph>` is the canonical emphasis element in this draft.

### Articulation and Phonation

Articulation and phonation elements mark delivery modes that are not reducible to dimensional prosody.

```xml
<elongated>
<pressured>
<stutter>
<whisper>
<murmur>
```

| Element | Meaning |
|---------|---------|
| `<elongated>` | drawn-out articulation; stretched vowel, syllable, word, or phrase |
| `<pressured>` | rushed, compressed, forced-through, urgent, anxious, or overfull articulation |
| `<stutter>` | disfluent, repeated, blocked, or interrupted articulation |
| `<whisper>` | whispered phonation; not necessarily quiet |
| `<murmur>` | soft, indistinct, partially articulated, under-the-breath, or low-audibility delivery |

`<elongated>` and `<pressured>` form an articulation-rate/pressure opposition.

`<whisper>` MUST NOT be treated as a subtype of `<quiet>`. Whispering is a phonation mode and MAY compose with loudness, harshness, gentleness, or other dimensions.

Example:

```xml
<say><loud><harsh><whisper>Get out.</whisper></harsh></loud></say>
```

`<murmur>` is a composite convenience element rather than a pure primitive dimension. Evaluators MAY interpret it as bundling low audibility, softened articulation, indistinctness, and/or detachment.

### Temporal and Breath Events

Temporal and breath events mark performed interruptions, silences, breaths, or breath-holding.

```xml
<pause>
<breath>
```

`<pause>` SHOULD use qualitative descriptors rather than numeric duration in core CNML.

Examples:

```xml
<pause kind="beat"/>
<pause kind="silence" quality="heavy"/>
<pause kind="hesitation" quality="nervous"/>
<pause quality="stunned"/>
```

Numeric or metrical timing belongs in more explicit temporal/music/layout systems, not in basic narrative prosody.

`<breath>` SHOULD use `direction` for the inhale/exhale/held axis.

Examples:

```xml
<breath direction="inhale"/>
<breath direction="exhale"/>
<breath direction="held"/>
<breath direction="exhale" quality="shaky">hahh</breath>
```

`type` SHOULD NOT be used for breath direction, because it is too generic.

### Nonlexical Vocal Events

Nonlexical vocal events represent expressive vocalizations that may or may not have explicit surface text.

```xml
<laugh>
<sob>
<sigh>
```

These elements MAY be empty when no literal surface vocalization is specified. When a surface vocalization is present, it SHOULD appear as the element content.

Examples:

```xml
<say>I'm fine. <laugh/> Really.</say>
<say>I'm fine. <laugh kind="chuckle">Heh.</laugh> Really.</say>
<sob intensity="high">I can’t—</sob>
<sigh quality="relieved">oh.</sigh>
```

Subtypes such as chuckle, giggle, cackle, snicker, or guffaw SHOULD be represented with attributes rather than separate canonical tags unless a later vocabulary extension deliberately promotes them.

Example:

```xml
<laugh kind="chuckle" intensity="low">Heh.</laugh>
```

## Information Integrity

Information-integrity elements mark reduced access to represented content, text, speech, inscription, memory, signal, perception, or rendered material.

CNML uses one canonical integrity element:

```xml
<obscured>
```

`<obscured>` indicates that the represented content is not fully accessible to the reader, listener, viewer, narrator, character, evaluator, or projection layer.

Obscuration is a broad operation. It may include corruption, degradation, erosion, redaction, censorship, occlusion, omission, substitution, perceptual distortion, memory loss, transmission loss, or other forms of reduced access.

Different forms of obscuration are distinguished by attributes rather than by separate canonical tags.

### Attributes

| Attribute | Meaning |
|-----------|---------|
| `method` | how access is reduced or what operation dominates |
| `source` | origin, agent, layer, force, medium, or mechanism responsible for the obscuration |
| `reason` | why the content is obscured, especially for censorship, redaction, suppression, or omission |
| `style` | author-declared surface aesthetic or corruption/concealment style |
| `quality` | qualitative modifier of the obscuration |
| `audio-effect` | renderer-facing hint for audio projection |

`method` identifies the dominant operation. Recommended `method` values include, but are not limited to:

```text
corruption
redaction
censorship
occlusion
omission
substitution
weathering
erosion
erasure
burning
perceptual_warp
memory_loss
transmission_loss
signal_loss
```

`source` identifies the origin, agent, layer, force, medium, or mechanism responsible for the obscuration. Recommended `source` values include, but are not limited to:

```text
weathering
erosion
fire
transmission
archive
memory
narrator
editor
publisher
platform
government
broadcast
machine
translation
encoding
eldritch
unknown
```

`reason` identifies why the content is obscured. Recommended `reason` values include, but are not limited to:

```text
profanity
privacy
classification
legal
copyright
spoiler
platform_policy
authorial
forbidden_knowledge
unknown
```

`style` identifies the author-declared surface aesthetic or corruption/concealment style. CNML does not validate that the authored surface form matches the declared style. Recommended `style` values include, but are not limited to:

```text
glitch
zalgo
mojibake
eldritch
fragmented
eroded
charred
illegible
static
ascii_noise
black_box
careless
partial
```

`audio-effect` is a renderer-facing hint for audio projection. It does not define the exact distortion algorithm. Exact auditory realization is renderer-defined. Recommended `audio-effect` values include, but are not limited to:

```text
bleep
silence
noise
dropout
mute
tone
garble
reverse
distort
static
```

### Author-Supplied Surface and Renderer Projection

For corruption-like obscuration, the author SHOULD provide the intended visible surface form directly in the source text.

Renderers SHOULD NOT generate visible glitch, mojibake, zalgo, distortion, or other corruption aesthetics unless an explicit renderer policy or non-core extension requests such projection.

For redaction-, censorship-, masking-, substitution-, or projection-dependent obscuration, the renderer MAY obscure, replace, mask, silence, bleep, or otherwise project the underlying content according to medium, geometry, timing, policy, and supplied attributes.

`style` metadata may be used by evaluators, indexers, or audio renderers to interpret the author's authored surface style or to choose projection behavior, but CNML core does not require visual validation against that style.

### Examples

```xml
<obscured method="corruption" source="transmission" style="glitch" audio-effect="static">
  I c█n st—ll h̴e̷a̵r̷ ̶y̶o̸u̴
</obscured>

<obscured method="corruption" source="archive" style="mojibake" audio-effect="garble">
  ॕൔᡆᔭጸ ٴѡࡳѴݥᡳᄠ
</obscured>

<obscured method="weathering" source="erosion" style="eroded">
  ELI--A M--R-- 18--–19--
</obscured>

<obscured method="perceptual_warp" source="eldritch" style="eldritch" audio-effect="distort">
  the true name
</obscured>

<obscured method="redaction" reason="classification" source="government" style="black_box">
  Operation Marigold
</obscured>

<obscured method="censorship" reason="profanity" source="broadcast" audio-effect="bleep">
  fuck
</obscured>
```

## Constraints

- CNML documents MUST be well-formed XML.
- Inline elements compose recursively.
- Structural frames MUST satisfy their legal immediate container rules.
- A context MUST NOT mix structural frames of different hierarchy ranks.
- Block content is lower-rank than structural frames for Same-Context Rank Discipline.
- Raw character data belongs inside text-bearing block or inline elements; structural frames SHOULD NOT contain raw character data directly except for insignificant whitespace.
- `<part>` is a high-level mainmatter structural frame.
- `<section>` structures frontmatter/backmatter section trees.
- `narrator` and `voice` are inherited by descendant content unless overridden by a nearer descendant value.
- `n` is the authored CNML address-segment attribute.
- `<ms/>` is the canonical zero-width source-position milestone.
- `.` separates address segments.
- `--` separates span endpoints.
- `(...)` compresses shared-prefix spans.
- `%XX` escapes reserved characters inside address expressions.
- `_` prefixes derived, non-authorial address segments.
- `xml:id` remains optional and XML-native; it does not replace CNML path addressing.
- Derived sentence, word, and token addresses require declared segmentation behavior before they can be considered reproducible.
- Cross-frame spans are outside required alpha processor behavior unless a referencing layer defines them.
- Whitespace-separated attribute token lists SHOULD be used when one attribute needs multiple qualitative values, as with `<express with="arms hands">` and `<say affect="homesickness protectiveness love">`.
- `affect` values SHOULD be composed from noun-form tokens rather than adjective-form descriptors.
- `lang` on `<say>` SHOULD use BCP 47-style language tags where possible, including standard subtags for dialect, region, script, and variant where available.
- `system` on `<say>` SHOULD use ISO 15924 script identifiers where applicable; notation, transcription, glossing, sign-language, or CNML-specific systems not covered by ISO 15924 MAY use evaluator-defined tokens.
- Signed languages are represented with `<say lang="..." system="...">`.
- `<c4>` is the preferred form for CNML-native C4/Purple Herring relation statements.
- In `<c4>`, bare `#` resolves to the nearest enclosing structural frame resource by default.
- In `<c4>`, `+{ancestor}#` explicitly selects the nearest enclosing ancestor of the named element or evaluator-recognized ancestor type.
- `<data lang="c4">` MUST NOT be assumed equivalent to `<c4>` unless an evaluator explicitly grants it CNML-native C4 resource access semantics.
- Generic `<data lang="...">` blocks do not imply C4/CNML-native resource access unless an adapter defines such mapping.
- `<obscured>` is the canonical information-integrity element for reduced access to represented content.
- `method`, `source`, `reason`, `style`, `quality`, and `audio-effect` on `<obscured>` are qualitative metadata or projection hints, not numeric controls.
- CNML does not validate that authored obscured surface text matches the declared `style`.
- Core narrative CNML SHOULD prefer qualitative expressive descriptors over numeric performance quantities unless an explicit temporal, music, or layout subsystem is in scope.
