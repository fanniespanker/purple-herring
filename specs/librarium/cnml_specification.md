# CNML Specification v0.5.2 (Draft)

This Purple Herring CNML draft incorporates the current CNML structural-frame, addressing/milestone, discourse, linguistic-system, prosody, affect, fish/C4 integration, and information-integrity revisions.

## Foundations

CNML (Contextual Narrative Markup Language) is a constrained XML vocabulary with semantic rules.

All CNML documents MUST be well-formed XML.

CNML is primarily an experiential and discourse-oriented representation system.

Experiential content refers to represented subjective or interpretive states, not objective truth claims.

CNML encodes semantic, experiential, structural, temporal, prosodic, and spatial relationships, not canonical presentation.

Evaluation, rendering, playback, and projection are interpreter-dependent realizations of CNML source structure.

Operational interpretation MAY derive additional structure for rendering or execution purposes, but such derived structure MUST NOT be treated as intrinsic source semantics.

---

## Core Evaluation Model

CNML evaluation operates across four primary expressive planes:

```text
Narrative × Temporal × Prosodic × Spatial
```

These planes are independently representable and compositional.

| Plane | Meaning |
|---|---|
| Narrative | discourse, cognition, experience |
| Temporal | musical/performance progression |
| Prosodic | affective modulation |
| Spatial | typographic/spatial arrangement |

---

## Discourse Epistemics

CNML discourse does not represent objective reality.

Discourse content is interpreted as a projection generated by a narrator, speaker, experiencer, evaluator, or other discourse context.

Discourse contexts MAY be unreliable, dishonest, misleading, hallucinating, delusional, temporally inconsistent, or partially reconstructive.

CNML discourse is therefore epistemically unstable by default.

## Root Element

```xml
<cnml>
```

The root element MAY contain structural frames, metadata, poems, music, discourse content, CNML-native fish relation blocks, and generic embedded data blocks.

## Structural Frames

Structural frames partition CNML source content into document, publication/work, matter, and evaluation scopes.

A structural frame identifies a source region and establishes containment.

A structural frame MUST NOT be interpreted, by itself, as asserting in-world truth, chronological order, narrative reliability, experiential modality, semantic equivalence, publication-resource identity, or rendering style.

Such meanings belong to explicit discourse, experiential, semantic, publication, rendering, or external graph constructs.

### Core Structural Frames

CNML defines the following structural frame elements:

```text
cnml
series
work
frontmatter
mainmatter
backmatter
part
act
prologue
interlude
epilogue
chapter
scene
section
```

### Structural Frame Roles

`<cnml>` is the root CNML container and establishes the document-level CNML processing context.

`<series>` is a publication-level container grouping one or more work-level containers.

`<work>` is the work-level source/publication frame. The `format` attribute identifies the work form, medium, or publication category.

Examples:

```xml
<work format="novel" n="Example Novel">
  ...
</work>

<work format="short_story" n="Example Story">
  ...
</work>
```

Recommended `format` values include, but are not limited to:

```text
novel
novella
book
short_story
story
poem
poetry_collection
essay
essay_collection
play
screenplay
serial
collection
anthology
comic
graphic_novel
interactive_fiction
```

`<frontmatter>` contains material preceding the main matter.

`<mainmatter>` contains the main work-content matter frame and the primary progression sequence of the work.

`<backmatter>` contains material following the main matter.

`<part>` is an optional high-level subdivision of `<mainmatter>`. It is above ordinary progression units such as acts, chapters, and scenes. It is not a generic segmentation element. Use `<part>` when the main matter has major authorial divisions above acts or chapters, such as named parts, books-within-books, movements, large thematic divisions, or macrosections.

`<act>` is a dramatic or authorial macro-division within `<mainmatter>` or `<part>`.

`<prologue>`, `<interlude>`, and `<epilogue>` are special mainmatter structural frames for authorial prefatory, intervening, or concluding material within the main work progression.

`<chapter>` is a major work subdivision within `<mainmatter>`, `<part>`, or `<act>`.

`<scene>` is a local source subdivision within `<mainmatter>`, `<part>`, `<act>`, `<prologue>`, `<interlude>`, `<epilogue>`, or `<chapter>`.

`<section>` is a paratextual structural subdivision for `<frontmatter>` and `<backmatter>`.

## Publication Metadata and Fish Relations

Publication frames MAY contain basic bibliographic/source metadata directly in CNML.

Recommended direct metadata elements include:

```text
title
author
date
identifier
license
summary
synopsis
```

Relational, ontological, interpretive, subject, genre, audience, adaptation, parody, support/opposition, portrayal, and thematic metadata SHOULD be represented with fish relation statements.

Example:

```xml
<work format="novel" n="Example Work" resource="https://example.org/work/example">
  <title>Example Work</title>
  <author>Example Author</author>
  <date of="publication">2025</date>
  <identifier scheme="IRI">https://example.org/work/example</identifier>

  <fish>
    #&has_genre@Fiction/noir;
    #&intended_audience@Audience/general;
    #Act 1.Chapter 1&portrays@Relationships/dynamics/example_dynamic;
    #Act 2.Chapter 3&explores@Ideas/example_theme;
  </fish>

  <mainmatter>
    <chapter n="Chapter 1">
      <scene n="Opening Scene">
        <p>...</p>
      </scene>
    </chapter>
  </mainmatter>
</work>
```

## Structural Container Rules

Each structural frame has a defined set of legal immediate containers.

A structural frame is conforming when its immediate parent is one of its legal containers and the containing context satisfies Same-Context Rank Discipline.

The root `<cnml>` MAY contain any structural frame as an immediate child. This permits CNML documents to represent structural fragments such as a standalone chapter, scene, or section.

A structural frame appearing directly under `<cnml>` is interpreted as the root frame of the current CNML fragment. Missing ancestors are not implied unless supplied by metadata, evaluator context, or an external referencing layer.

| Structural frame | Legal immediate containers |
|---|---|
| `<series>` | `<cnml>` |
| `<work>` | `<cnml>`, `<series>` |
| `<frontmatter>` | `<cnml>`, `<work>` |
| `<mainmatter>` | `<cnml>`, `<work>` |
| `<backmatter>` | `<cnml>`, `<work>` |
| `<part>` | `<cnml>`, `<mainmatter>` |
| `<act>` | `<cnml>`, `<mainmatter>`, `<part>` |
| `<prologue>` | `<cnml>`, `<work>`, `<mainmatter>`, `<part>`, `<act>` |
| `<interlude>` | `<cnml>`, `<work>`, `<mainmatter>`, `<part>`, `<act>` |
| `<epilogue>` | `<cnml>`, `<work>`, `<mainmatter>`, `<part>`, `<act>` |
| `<chapter>` | `<cnml>`, `<mainmatter>`, `<part>`, `<act>`, `<prologue>`, `<interlude>`, `<epilogue>` |
| `<scene>` | `<cnml>`, `<mainmatter>`, `<part>`, `<act>`, `<prologue>`, `<interlude>`, `<epilogue>`, `<chapter>` |
| `<section>` | `<cnml>`, `<frontmatter>`, `<backmatter>`, `<section>` |

When `<work>` directly contains `<prologue>`, `<interlude>`, `<epilogue>`, `<part>`, `<act>`, `<chapter>`, `<scene>`, or block content, those children are interpreted as implicit-mainmatter content unless matter frames are also present in the same context.

### Same-Context Rank Discipline

A context MUST NOT mix structural frames of different hierarchy ranks.

A structural frame may appear directly in a context only when no higher-rank structural frame appears in that same context.

If a context contains a higher-rank frame, lower-rank frames and block content MUST appear inside the appropriate descendant frame rather than as siblings of the higher-rank frame.

This permits structural fragments while preserving the CNML frame hierarchy.

### Mainmatter Structural Sequence

The elements `<part>`, `<act>`, `<prologue>`, `<interlude>`, `<epilogue>`, `<chapter>`, and `<scene>` are mainmatter structural frames.

The ordinary progression rank order is:

```text
part → act → chapter → scene
```

`<prologue>`, `<interlude>`, and `<epilogue>` are special mainmatter frames that may occur at mainmatter, part, or act scope.

### Sections

Sections are exclusively for structuring `<frontmatter>` and `<backmatter>`.

Sections MAY nest within `<frontmatter>`, `<backmatter>`, or another `<section>` that is itself contained within frontmatter or backmatter.

### Frame Headers

The following structural frames MAY contain zero or one of each frame-header element permitted by the active CNML profile:

```text
series
work
section
part
act
prologue
interlude
epilogue
chapter
scene
```

Common frame-header elements include:

```text
title
summary
synopsis
subheading
```

The `n` attribute and `<title>` serve different roles. `n` supplies an address segment; `<title>` supplies title/display metadata.

### Content Containers

Structural-frame parentage defines where structural frames may appear.

Text-bearing source content is represented by block content and inline content, not by structural frames.

Block content includes paragraphs, expression elements, experiential modality elements, poem/music blocks, fish blocks, semantic extension points, and other source-content elements permitted by the active CNML profile.

Block content MAY appear inside:

```text
cnml
frontmatter
mainmatter
backmatter
part
act
prologue
interlude
epilogue
chapter
scene
section
```

A block-content element appearing directly under `<cnml>` is interpreted as a content fragment.

Block content is lower-rank than structural frames for Same-Context Rank Discipline.

If a context contains a structural frame, block content MUST appear inside the appropriate descendant frame rather than as a sibling of that structural frame.

Raw character data belongs inside text-bearing block or inline elements.

Inline content includes text nodes, milestones, information-integrity elements, prosody elements, and other inline elements permitted by the active CNML profile.

Structural frames SHOULD NOT contain raw character data directly except for insignificant whitespace.

Text-bearing block and inline elements define their own mixed-content behavior.

## Narrative Voice Context

CNML defines `narrator` and `voice` as optional inheritable discourse-context attributes.

These attributes MAY appear on structural frames and discourse containers.

`narrator` identifies the discourse source, narrator identity, persona, or narrative agent associated with the content.

`voice` identifies narrative voice, register, style, tone, rhetorical mode, or discourse channel.

`narrator` and `voice` are inherited by descendant content unless overridden by a nearer descendant value.

Inheritance is nearest-ancestor inheritance:

```text
effective(narrator, node) = nearest ancestor-or-self value of @narrator
effective(voice, node)    = nearest ancestor-or-self value of @voice
```

Expression-specific attributes such as `who` identify represented speakers, thinkers, feelers, or experiencers. They do not replace inherited narration context unless the expression element also explicitly sets `narrator`.

## Addressing, Milestones, and Derived Names

This section defines CNML-local addressing over authored `n` values, milestones, span expressions, grouped spans, derived address segments, and resource-internal subresource path behavior shared with Purple Herring where applicable.

### Local CNML Names

CNML elements MAY carry an `n` attribute.

The `n` attribute defines an authored local CNML name usable as a segment in CNML address expressions.

```xml
<work format="novel" n="Example Work">
  <act n="Act 1">
    <chapter n="Chapter 4">
      <p>
        <ms n="ms5"/>Text.<ms n="ms9"/>
      </p>
    </chapter>
  </act>
</work>
```

The milestone may be addressed as:

```text
Example Work.Act 1.Chapter 4.ms5
```

`n` is not an XML identifier. It is not required to be document-global.

`xml:id` MAY additionally be used when XML-native document-global identity is required.

`resource` MAY be used separately for persistent semantic or publication identity.

| Attribute | Meaning |
|---|---|
| `n` | CNML local address segment |
| `xml:id` | XML document-global node identity |
| `resource` | persistent semantic/publication resource identity |

### CNML Address Expressions

A CNML address expression is a path over authored `n` values and/or processor-derived address segments.

The path separator is `.`.

Each unescaped `.` separates one path segment from the next.

A path segment resolves against an addressable CNML unit in the current address scope.

```ebnf
address = segment, { ".", segment } ;
```

Whitespace MAY appear literally in `n` values and in CNML address expressions.

### Escaping in Address Expressions

Characters or sequences with syntactic meaning in CNML address expressions MUST be escaped when they occur literally inside an `n` value.

CNML address expressions use percent escapes.

Reserved syntax:

```text
.      path separator
--     span separator
%      escape introducer
( )    grouping delimiters
```

Required escapes include:

```text
.   => %2E
%   => %25
(   => %28
)   => %29
-   => %2D when needed to prevent `--`
```

Percent escapes are interpreted during address-expression parsing, after XML entity processing and before segment comparison.

### Milestones

CNML provides the empty `<ms/>` element as a source-position milestone.

A milestone marks an addressable position in an ordered CNML source stream.

A milestone:

1. MUST be empty.
2. MAY carry an authored `n`.
3. MAY carry an `xml:id` for XML-native identity.
4. Marks a position, not a content container.
5. Has no default rendering.
6. Does not itself define a span.
7. Is semantically neutral unless referenced by another construct.

Example:

```xml
<p>
  <ms n="start"/>The old house <emph>remembered</emph> her.<ms n="end"/>
</p>
```

Milestones MUST NOT produce visible output by default.

Renderers MAY expose milestones in debugging, editorial, or annotation views.

### Span Expressions

A CNML span expression identifies an ordered source interval between two addresses.

The span separator is `--`.

```text
start-address--end-address
```

A milestone-delimited span contains all source content after the start milestone and before the end milestone.

The milestone elements themselves are boundary markers and are not part of the span payload.

Within a single source stream, the start address of a span MUST precede or equal the end address.

If both addresses resolve to the same source position, the resulting span is empty.

Reverse spans are not defined by CNML alpha.

### Grouped Shared-Prefix Spans

A span whose endpoints share a prefix MAY be written using grouped shared-prefix syntax.

```text
prefix.(start--end)
```

Example:

```text
Example Work.Act 1.Chapter 4.(ms5--ms9)
```

Grouped spans are authoring sugar. Processors MAY normalize them to expanded span expressions.

For alpha, grouped spans MUST NOT contain nested grouped expressions.

### Overlapping XML Structure

Milestone-delimited spans MAY overlap XML element boundaries.

This permits CNML to address source intervals that cannot be represented by ordinary XML containment without altering the document tree.

### Structural Boundary Constraint

A locally conforming milestone-delimited span SHOULD NOT cross structural frame boundaries unless a referencing layer explicitly defines cross-frame span semantics.

A milestone pair is locally conforming when both milestones occur within the same nearest structural frame.

Structural frames include:

```text
series
work
frontmatter
mainmatter
backmatter
section
part
act
prologue
interlude
epilogue
chapter
scene
```

Referencing layers MAY define cross-frame behavior, but CNML alpha processors are not required to support it.

### Address Scopes

CNML addresses are evaluated relative to an address scope.

Structural elements with `n` values establish addressable path segments.

The following elements SHOULD be treated as addressable units when present:

```text
series
work
frontmatter
mainmatter
backmatter
section
part
act
prologue
interlude
epilogue
chapter
scene
poem
music
stanza
line
p
ms
```

Derived sentence, word, and token addresses are derived pseudo-unit addresses. They do not imply canonical `<s>`, `<w>`, or `<t>` elements in source.

A segmentation profile is the declared rule set used by a processor to derive sentence, word, or token boundaries from source text.

Derived sentence, word, and token pseudo-unit addresses are reproducible only when the segmentation profile is declared or otherwise fixed.

### Derived Address Segments

Any addressable CNML unit without an authored `n` MAY be assigned a processor-derived address segment.

Derived address segments MUST begin with `_`.

Derived segments are non-authorial. They are not intrinsic source data.

Authored `n` values beginning with `_` are reserved and SHOULD NOT be used in conforming CNML documents.

Explicit `n` values define authored address segments and take precedence over derived address segments.

Recommended derived segment forms:

```text
_series1       first series
_work1         first work
_frontmatter1  first frontmatter
_mainmatter1   first mainmatter
_backmatter1   first backmatter
_sec1          first section
_part1         first part
_act1          first act
_prologue1     first prologue
_interlude1    first interlude
_epilogue1     first epilogue
_ch1           first chapter
_sc1           first scene
_p1            first paragraph
_l1            first line
_st1           first stanza
_s1            first derived sentence pseudo-unit
_w1            first derived word pseudo-unit
_t1            first derived token pseudo-unit
_ms1           first anonymous or generated milestone
```

Element-kind ordinals are counted by element kind within the relevant containing scope.

Derived address segments are positional and may change when the source changes.

Sentence, word, and token segmentation are profile-dependent unless explicitly marked by a profile-defined mechanism.

Milestones SHOULD be used for durable fine-grained addressing.

### Resource and Subresource Paths

CNML address expressions may be used as subresource paths inside larger resource references.

A resource identity and a path inside that resource SHOULD remain separable.

Recommended conceptual form:

```text
resource#subresource-path
```

Within Purple Herring or another semantic layer, a CNML subresource path SHOULD be embedded or quoted as a resource-internal path rather than reinterpreted as an ontology path.

Example:

```text
cnml:"Example Work#Act 1.Chapter 4.(ms5--ms9)"
```

CNML and Purple Herring SHOULD share the subresource path grammar where both are addressing positions inside concrete resources.

This does not require CNML source paths and PH semantic/concept traversal paths to have identical semantics.

### Summary of Address Syntax

```text
.      path/subresource separator
--     span separator
(...)  grouped shared-prefix expression
%XX    percent escape
_      derived segment prefix
#      resource/subresource boundary, where used by a referencing layer
```

## Fish Relations and Embedded Data

CNML distinguishes CNML-native fish relation blocks from generic embedded data blocks.

`<fish>` is tightly coupled to CNML and receives CNML-aware resource access semantics.

`<data format="...">` embeds generic non-CNML-native data or serialization formats. Such blocks are parsed according to their declared format and do not receive fish/CNML resource access semantics by default unless an evaluator or adapter explicitly defines such mapping.

### Fish Blocks

```xml
<fish>
```

`<fish>` embeds Purple Herring / C4 relation statements whose resource-resolution context is the surrounding CNML document.

A `<fish>` element MAY contain one or more fish statements. The element content may be parsed as a C4 school fragment when multiple statements are present.

A `<fish>` block MAY address CNML-native resources such as the current structural frame resource, declared anchors or milestone spans, `xml:id`-addressable elements, and evaluator-defined CNML projections.

Inside `<fish>`, bare `#` denotes the current structural frame resource.

The current structural frame resource is the nearest enclosing ancestor CNML element belonging to the structural-frame set.

The structural-frame set is:

```text
series
work
frontmatter
mainmatter
backmatter
section
part
act
prologue
interlude
epilogue
chapter
scene
```

By default, `#` MUST NOT resolve to discourse, expression, prosody, integrity, data, or other non-structural elements merely because the `<fish>` block is nested inside them.

### Explicit Ancestor Selector

The explicit ancestor selector has the form:

```text
+{ancestor}#
```

`+{ancestor}#` overrides default `#` resolution by walking upward from the current `<fish>` block to the nearest enclosing ancestor whose element name or evaluator-recognized ancestor type is `{ancestor}`.

Authors SHOULD use bare `#` when referring to the current structural frame, and `+{ancestor}#` when referring to a non-default ancestor or to a more distant structural ancestor.

Fish resource access semantics are CNML-integrated. They are not equivalent to RDF URI expansion, JSON-LD context processing, Turtle prefix handling, or other external relation-format semantics.

### Generic Data Blocks

```xml
<data format="...">
```

`<data>` embeds external data, metadata, relation languages, or serialization formats. The `format` attribute is REQUIRED and identifies the embedded language, notation, or serialization format.

`<data format="fish">` MAY be accepted as a generic embedded-data form, but it is not equivalent to `<fish>` unless an evaluator explicitly grants it the same CNML-native resource access semantics. Authors SHOULD use `<fish>` when Purple Herring/C4 is intended to operate as the CNML-native semantic relation layer.

## Expression Modalities

Expression modalities mark the kind of discourse-content being represented. They mark what communicative or experiential channel carries the content, not necessarily the literal surface action that caused it.

```xml
<say lang="..." system="...">
<think>
<feel>
<express with="...">
```

| Element | Meaning |
|---|---|
| `<say>` | externalized linguistic discourse |
| `<think>` | propositional/internal language |
| `<feel>` | autonomic, affective, limbic, or embodied cognition |
| `<express with="...">` | semantic content conveyed through nonlinguistic expressive channels such as body, hands, face, gaze, posture, stillness, or silence |

`<say>` SHOULD be used for externalized linguistic discourse, including spoken language, written language, signed language, and other language-bearing systems. The `lang` and `system` attributes identify the represented language and the surface notation, script, transcription system, or encoding used in the element content.

`lang` SHOULD use BCP 47-style language tags where possible.

`system` SHOULD use ISO 15924 script identifiers where applicable. When the content is encoded using a notation, transcription system, glossing convention, sign-language writing system, or CNML-specific DSL not covered by ISO 15924, `system` MAY use an evaluator-defined token.

For signed languages, facial expression, gaze, posture, and body movement that are grammatical features of the signed language are interpreted according to the declared `system`. They SHOULD NOT be separately represented with `<express>` unless the author intends an additional nonlinguistic expression outside the signed utterance.

`<express>` SHOULD be used for semantic content communicated through nonlinguistic expressive channels.

The `with` attribute identifies one or more qualitative expression channels.

Recommended core `with` tokens include face, eyes, gaze, mouth, brows, head, hand, hands, finger, arms, shoulders, posture, body, movement, stillness, and silence.

### Affect Attribute

Expression modality elements MAY carry an `affect` attribute.

`affect` marks the affective components coloring or charging the represented expression, thought, feeling, or nonlinguistic expression. It is not identical to prosody: affect describes emotional composition; prosody describes performed quality, delivery, articulation, or timing.

`affect` is a whitespace-separated token list of noun-form affect components. Multiple tokens indicate co-present affective components rather than ordered sequence, ranking, or intensity.

Affect tokens SHOULD be nouns or noun-like nominal forms, not adjectives.

## Prosody

Prosody is a compositional modulation system. Prosodic elements modify expression quality, articulation, phonation, or performed timing. Prosody does not alter discourse structure.

Prosodic elements MAY appear inside any text-bearing CNML element whose content model permits inline/prosodic content, including paragraphs, titles, summaries, synopses, expression modalities, experiential modalities, information-integrity elements, and other text-bearing block or inline elements.

Prosody modifies expression quality, does not alter discourse structure, composes recursively, and is evaluator-interpreted.

Prosodic elements MAY also carry `affect` when the affective coloring applies specifically to the prosodic modifier rather than the whole expression.

### Dimensional Prosody

Dimensional prosody consists of opposed, composable expressive-quality dimensions.

```xml
<quiet>     <loud>
<calm>      <excited>
<gentle>    <forceful>
<smooth>    <harsh>
<engaged>   <detached>
<emph>      <deemph>
```

| Pair | Axis |
|---|---|
| `<quiet>` / `<loud>` | volume, projection, acoustic intensity |
| `<calm>` / `<excited>` | arousal |
| `<gentle>` / `<forceful>` | affective pressure, interpersonal force |
| `<smooth>` / `<harsh>` | delivery texture |
| `<engaged>` / `<detached>` | affective or interpersonal involvement |
| `<emph>` / `<deemph>` | stress or de-stress |

`<emph>` and `<deemph>` are prosodic stress/de-stress elements, not typographic formatting elements.

### Articulation and Phonation

Articulation and phonation elements mark delivery modes that are not reducible to dimensional prosody.

```xml
<elongated>
<pressured>
<stutter>
<whisper>
<murmur>
```

`<whisper>` MUST NOT be treated as a subtype of `<quiet>`. Whispering is a phonation mode and MAY compose with loudness, harshness, gentleness, or other dimensions.

### Temporal and Breath Events

Temporal and breath events mark performed interruptions, silences, breaths, or breath-holding.

```xml
<pause>
<breath>
```

`<pause>` SHOULD use qualitative descriptors rather than numeric duration in core CNML.

`<breath>` SHOULD use `direction` for the inhale/exhale/held axis.

### Nonlexical Vocal Events

Nonlexical vocal events represent expressive vocalizations that may or may not have explicit surface text.

```xml
<laugh>
<sob>
<sigh>
```

These elements MAY be empty when no literal surface vocalization is specified. When a surface vocalization is present, it SHOULD appear as the element content.

Subtypes such as chuckle, giggle, cackle, snicker, or guffaw SHOULD be represented with attributes rather than separate canonical tags unless a later vocabulary extension deliberately promotes them.

## Music and External Notation Systems

`<music>` is a CNML content element for embedded or referenced musical/performance material.

`<music>` uses `format` to identify the notation or serialization format.

Examples:

```xml
<music format="cinnamon">...</music>
<music format="musicxml">...</music>
<music format="midi">...</music>
<music format="abc">...</music>
```

CiNnaMoN is a separate music/performance notation system. It may be embedded in CNML with `format="cinnamon"`, but its grammar is not intrinsic to CNML.

PÂTE is a separate typographic/spatial layout system. CNML may embed or reference PÂTE through an explicit format-bearing extension point, but PÂTE's grammar is not intrinsic to CNML.

## Information Integrity

Information-integrity elements mark reduced access to represented content, text, speech, inscription, memory, signal, perception, or rendered material.

CNML uses one canonical integrity element:

```xml
<obscured>
```

`<obscured>` indicates that the represented content is not fully accessible to the reader, listener, viewer, narrator, character, evaluator, or projection layer.

Obscuration is a broad operation. It may include corruption, degradation, erosion, redaction, censorship, occlusion, omission, substitution, perceptual distortion, memory loss, transmission loss, signal loss, or other forms of reduced access.

Different forms of obscuration are distinguished by attributes rather than by separate canonical tags.

| Attribute | Meaning |
|---|---|
| `method` | how access is reduced or what operation dominates |
| `source` | origin, agent, layer, force, medium, or mechanism responsible for the obscuration |
| `reason` | why the content is obscured, especially for censorship, redaction, suppression, or omission |
| `style` | author-declared surface aesthetic or corruption/concealment style |
| `quality` | qualitative modifier of the obscuration |
| `audio-effect` | renderer-facing hint for audio projection |

For corruption-like obscuration, the author SHOULD provide the intended visible surface form directly in the source text.

Renderers SHOULD NOT generate visible glitch, mojibake, zalgo, distortion, or other corruption aesthetics unless an explicit renderer policy or non-core extension requests such projection.

For redaction-, censorship-, masking-, substitution-, or projection-dependent obscuration, the renderer MAY obscure, replace, mask, silence, bleep, or otherwise project the underlying content according to medium, geometry, timing, policy, and supplied attributes.

## Rendering Model

CNML MAY be rendered into HTML, PDF, TeX/LaTeX DVI, SVG, EPUB, Markdown, PostScript, OpenDocument, audio, or interactive runtimes.

### Recommended HTML Projection

| CNML | HTML |
|---|---|
| narrative units | `<p>` |
| prosody | `<span>` |
| poem/music | inline SVG or evaluator-defined projection |
| fish/data blocks | non-rendered metadata or evaluator-defined projection |
| obscured | author-supplied surface text and/or renderer-mediated obscuration according to method and projection policy |

## Constraints

- CNML documents MUST be well-formed XML.
- Inline elements compose recursively.
- Structural frames MUST satisfy their legal immediate container rules.
- A context MUST NOT mix structural frames of different hierarchy ranks.
- Block content is lower-rank than structural frames for Same-Context Rank Discipline.
- Raw character data belongs inside text-bearing block or inline elements; structural frames SHOULD NOT contain raw character data directly except for insignificant whitespace.
- `<work>` is the work-level source/publication frame; `format` identifies the work form, medium, or publication category.
- `<part>` is a high-level mainmatter structural frame.
- `<prologue>`, `<interlude>`, and `<epilogue>` are special mainmatter structural frames.
- `<section>` structures frontmatter/backmatter section trees.
- `narrator` and `voice` are inherited by descendant content unless overridden by a nearer descendant value.
- `n` is the authored CNML address-segment attribute.
- `<ms/>` is the canonical zero-width source-position milestone.
- `.` separates address segments.
- `--` separates span endpoints.
- `(...)` compresses shared-prefix spans.
- `%XX` escapes reserved characters inside address expressions.
- `_` prefixes derived, non-authorial address segments.
- `xml:id` remains optional and XML-native; it does not replace CNML path addressing.
- Derived sentence, word, and token addresses are pseudo-unit addresses and require declared segmentation behavior before they can be considered reproducible.
- `<s>`, `<w>`, and `<t>` are not canonical CNML elements in this draft.
- Cross-frame spans are outside required alpha processor behavior unless a referencing layer defines them.
- `<fish>` is the preferred form for CNML-native Purple Herring/C4 relation statements.
- In `<fish>`, bare `#` resolves to the nearest enclosing structural frame resource by default.
- In `<fish>`, `+{ancestor}#` explicitly selects the nearest enclosing ancestor of the named element or evaluator-recognized ancestor type.
- `<data format="fish">` MUST NOT be assumed equivalent to `<fish>` unless an evaluator explicitly grants it CNML-native fish resource access semantics.
- Generic `<data format="...">` blocks do not imply CNML-native resource access unless an adapter defines such mapping.
- `<music>` uses `format` to identify the embedded or referenced music/performance notation.
- CiNnaMoN and PÂTE are separate systems; CNML may embed or reference them, but their grammars are not intrinsic to CNML.
- `<obscured>` is the canonical information-integrity element for reduced access to represented content.
- `method`, `source`, `reason`, `style`, `quality`, and `audio-effect` on `<obscured>` are qualitative metadata or projection hints, not numeric controls.
- CNML does not validate that authored obscured surface text matches the declared `style`.
- `affect` values SHOULD be composed from noun-form tokens rather than adjective-form descriptors.
- Whitespace-separated attribute token lists SHOULD be used when one attribute needs multiple qualitative values.
- `lang` on `<say>` SHOULD use BCP 47-style language tags where possible.
- `system` on `<say>` SHOULD use ISO 15924 script identifiers where applicable; notation, transcription, glossing, sign-language, or CNML-specific systems not covered by ISO 15924 MAY use evaluator-defined tokens.
- Signed languages are represented with `<say lang="..." system="...">`.
- Core narrative CNML SHOULD prefer qualitative expressive descriptors over numeric performance quantities unless an explicit temporal, music, or layout subsystem is in scope.
